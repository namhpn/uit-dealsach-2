## Requirements

### 1. Frontend dependency audit and generated-output hygiene

- Run:

```bash
docker compose run --rm frontend npm install
docker compose run --rm frontend npm audit
```

* Identify the package or dependency chain responsible for the high-severity audit issue reported in T0001.
* Prefer the smallest safe dependency update that resolves the audit issue.
* Use npm only. Do not switch to pnpm, yarn, bun, or another package manager.
* Do not add new runtime dependencies unless required to resolve the audit issue and clearly reported.
* Do not use `npm audit fix --force` unless:

  * the resulting dependency changes are reviewed,
  * no major framework or architecture change is introduced,
  * `npm run build` still passes,
  * and the completion report explicitly explains why force was necessary.
* If the audit issue cannot be resolved safely within this ticket scope, keep dependencies unchanged and record the unresolved issue with exact audit details in `docs/Known_Issues_And_Followups.md`.

Generated-output hygiene:

* Inspect `.gitignore`.
* Verify whether `frontend/node_modules/` is already ignored.
* Add missing frontend generated-output rules only where absent.
* Required ignore coverage after this ticket:

  * `frontend/node_modules/`
  * `frontend/dist/`
* Avoid duplicate or conflicting ignore entries.
* Current GitHub `main` already includes `/frontend/node_modules`, so Codex must verify and avoid adding a duplicate entry. 
* If `.gitignore` already covers an item, update `docs/Known_Issues_And_Followups.md` to close or correct KI-0005 accurately.

### 2. Core schema

Create migrations for the following backend tables:

* `categories`
* `books`
* `retailer_platforms`
* `merchants`
* `offers`
* `observation_cycles`
* `price_observations`

The schema must support:

* A book has exactly one primary category.
* A retailer platform has many merchants.
* A merchant belongs to one retailer platform.
* An offer belongs to one book, one retailer platform, and one merchant.
* An offer may have many price observations.
* A price observation belongs to one offer and one observation cycle.
* Historical observations preserve observation-time facts and do not rely only on current entity state.

### 3. Required fields

#### Categories

* Name
* Slug or stable key
* Status: `active`, `archived`
* Created/updated timestamps

#### Books

* Title
* Author
* Publisher
* ISBN, nullable
* Description, nullable
* Cover image URL/path, nullable
* Primary category ID
* Featured flag
* Status: `active`, `archived`
* Created/updated timestamps

#### Retailer platforms

* Name
* Slug or stable key
* Approved domains as JSON or equivalent safe structure
* Status: `active`, `archived`
* Created/updated timestamps

#### Merchants

* Retailer platform ID
* Name
* Slug or stable key
* Status: `active`, `archived`
* Created/updated timestamps

#### Offers

* Book ID
* Retailer platform ID
* Merchant ID
* External offer title
* Affiliate destination URL, nullable
* Destination validity flag or status
* Offer status: `pending_review`, `active`, `unavailable`, `inactive`, `removed_invalid`
* Created/updated timestamps

#### Observation cycles

* Cycle date
* Processed-at timestamp using Vietnam-time semantics
* Notes, nullable
* Created/updated timestamps

#### Price observations

Each observation must store:

* Offer ID
* Observation cycle ID
* Observed-at timestamp
* Availability status
* Listed item price, nullable only when unavailable
* Book status at observation time
* Offer status at observation time
* Retailer status at observation time
* Merchant status at observation time
* Merchant-retailer consistency at observation time
* Destination validity at observation time
* Created/updated timestamps

### 4. Constraints and indexes

* Prices must be whole-number VND values greater than 0 when present.
* Offer statuses must align with the requirement status model.
* Category, book, retailer, and merchant lifecycle status fields must support active/archived semantics.
* Preserve dependent history; do not design hard-delete-dependent behavior.
* Add indexes for:

  * book/category
  * offer/book
  * offer/retailer
  * offer/merchant
  * observation/offer
  * observation/cycle
  * observation observed date/time
* Add database-level foreign keys where compatible with the existing CodeIgniter/MariaDB setup.

### 5. CodeIgniter models

Add or update models for:

* Category
* Book
* RetailerPlatform
* Merchant
* Offer
* ObservationCycle
* PriceObservation

Models must:

* Use CodeIgniter 4 model conventions.
* Define allowed fields.
* Define timestamps where applicable.
* Avoid raw SQL string interpolation.

### 6. Mock/demo seed data

Create deterministic seed data satisfying:

* At least 12 books.
* At least 4 retailer platforms.
* At least 8 merchants.
* At least 2 offers per book on average.
* At least 6 books with 14 days of price observations.
* At least 3 books with multi-retailer price history.
* At least 2 tied lowest-price scenarios.
* At least 3 price-drop scenarios.
* At least 2 unavailable-offer scenarios.
* At least 2 stale-price scenarios.
* At least 2 redirect-failure scenarios through invalid or missing destinations.

Seed data must be Vietnamese-first and suitable for Vietnamese users.

### 7. Tests

Add backend tests that verify:

* Migrations run successfully on a clean database.
* Seeds create the required minimum counts.
* Every book has exactly one category.
* Every merchant belongs to one retailer platform.
* Every offer belongs to one book, one retailer platform, and one merchant.
* Price observations store observation-time status fields.
* Seeded mock data includes:

  * 12+ books
  * 4+ retailers
  * 8+ merchants
  * average 2+ offers per book
  * 6+ books with 14 days of observations
  * 3+ price-drop scenarios
  * 2+ unavailable-offer scenarios
  * 2+ stale-price scenarios
  * 2+ redirect-failure scenarios
