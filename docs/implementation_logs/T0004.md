### 3. Backend affiliate Buy flow persistence

- Add database persistence for separate event concepts:
  - Buy Attempt
  - Affiliate Redirect
  - Redirect Failure
- Preserve event semantics:
  - Buy Attempt means the user clicked Buy.
  - Affiliate Redirect means DealSach validated the destination and issued the external redirect.
  - Redirect Failure means the Buy Attempt was blocked due to missing, invalid, unsafe, or otherwise ineligible destination.
- Store enough event metadata for demo/reporting integrity:
  - offer identifier
  - book identifier
  - retailer platform identifier
  - merchant identifier
  - event timestamp using `Asia/Ho_Chi_Minh`
  - event type/status
  - failure reason where applicable
  - safe destination domain/path summary where applicable
- Do not store full affiliate tracking parameters in an unmasked report/audit-friendly field.
- Use migrations and CI4 models/query builder. Do not use raw SQL string interpolation with user input.

### 4. Public Buy redirect endpoint

- Add a public Buy route, for example:

  ```text
  GET /go/offers/{offerId}
  ```

* The Buy route must be available to Guest Users and Registered Users.
* When the route is called for an existing offer, record a Buy Attempt before destination validation.
* Reuse or centralize the same current eligible-offer logic used by public catalog responses.
* Validate affiliate destinations:

  * must use `https://`
  * must match exact approved domain or approved subdomain
  * affiliate redirector domains are allowed only when explicitly listed in the retailer platform approved-domain list
* For a valid eligible destination:

  * record an Affiliate Redirect
  * count it as an affiliate click
  * issue the external redirect
* For missing, invalid, unsafe, stale, unavailable, inactive, removed, archived, merchant/retailer-inconsistent, or otherwise ineligible offers:

  * record a Redirect Failure when the offer is known
  * do not record an Affiliate Redirect
  * do not redirect externally
  * return a Vietnamese error page or response
* The route must not create carts, orders, payments, checkout sessions, shipping records, voucher logic, coupon logic, reviews, ratings, or direct-purchase behavior.

### 5. Popular clicked deals backed by persisted events

* Update `GET /api/public/discovery` so `popular_clicked_deals` uses successful Affiliate Redirect records.
* Ranking rules:

  * Use Affiliate Redirect records only.
  * Use the last 7 days in `Asia/Ho_Chi_Minh`.
  * Group by book identifier.
  * Sort by successful Affiliate Redirect count descending, then book title ascending.
* Each popular clicked deal card must include:

  * total Affiliate Redirect count in the last 7 days
  * top retailer platform by Affiliate Redirect count when available
* Keep the Vietnamese empty state only when no qualifying Affiliate Redirect records exist.
* Add deterministic seed data or test fixtures sufficient to verify popular clicked deals.
* Close `KI-0007` if fully resolved.
