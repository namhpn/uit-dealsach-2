## Requirements

### 1. Review and compare the current and original public UI

- Read `./AGENTS.md` before implementation.
- Compare the current frontend against `frontend-original/`, especially:
  - `HomePage.tsx`
  - `ProductDetailPage.tsx`
  - `shared.tsx`
- Identify which T0004 changes altered the original visual structure.
- Restore the original visual direction instead of replacing the pages with a new layout.
- Keep the T0004 API-backed behavior and do not restore mock catalog data as the runtime source of truth.

### 2. Restore homepage composition from the original design

Restore the original homepage feel and section structure from `frontend-original/src/app/pages/HomePage.tsx`, including the Neubrutalist visual direction:

- Deal/banner carousel or equivalent original hero/banner treatment.
- Featured book section.
- Recent price-drop section.
- Popular clicked-deals section.
- “How DealSach works” guidance.
- Dense, scannable book-card layout.
- Hard borders and hard shadows.
- Emerald primary surfaces.
- Red discount/action emphasis.
- Warm off-white/bone surfaces.
- Be Vietnam Pro typography.

The restored homepage must still use API data from:

- `GET /api/public/discovery`
- `GET /api/public/books` only if needed for supporting shelves or related sections
- `GET /api/public/filters` only if needed for category/filter metadata

Do not use the original mock arrays as homepage catalog data:

- `featuredCategories`
- `priceDropBooks`
- `popularDeals`

Static decorative banner copy or images may remain only if they do not drive catalog/book data and do not imply real-time pricing.

### 3. Preserve T0004 API-backed homepage behavior

- Featured books must come from `GET /api/public/discovery`.
- Recent price drops must come from `GET /api/public/discovery`.
- Popular clicked deals must come from `GET /api/public/discovery`.
- Popular clicked deal cards must still show:
  - 7-day successful Affiliate Redirect count
  - top retailer platform when available
- Recent price-drop cards must still show VND drop amount.
- Empty sections must show Vietnamese empty-state copy from the API when available.
- The homepage must include a prominent search entry point that navigates to `/search?q=...`.
- The homepage must clearly state that DealSach compares prices and redirects users to external sellers.
- The required public price disclaimer must appear near price-bearing areas:

```text
Giá tham khảo được ghi nhận gần đây, vui lòng kiểm tra lại tại nơi bán trước khi mua.
```

### 4. Restore book detail composition from the original design

Restore the original detail-page structure from `frontend-original/src/app/pages/ProductDetailPage.tsx` while replacing mock data with `GET /api/public/books/{bookId}`:

* Hero section with cover area and book metadata.
* Best/current lowest eligible price highlight.
* Wishlist visual affordance only as a login-required prompt or disabled placeholder if real auth is not implemented.
* Market/retailer price comparison list.
* Visually distinct unavailable and missing-link rows.
* Technical details block.
* Price history chart.
* Related/discovery book section if it can be populated from existing public APIs without backend changes.
* Disclaimer block near the end.

The restored detail page must continue to render:

* title
* author
* publisher
* category
* ISBN
* cover image or placeholder
* description
* lowest eligible price summary when available
* grouped offers:

  * purchasable
  * unavailable
  * stale reference
  * missing valid seller link
* book-level price history
* price disclaimer
* affiliate disclosure near Buy actions

### 5. Preserve Buy-flow behavior

* Purchasable offers must link to the DealSach Buy route supplied by the API, not directly to external seller URLs.
* Buy buttons must preserve T0004 behavior through `GET /go/offers/{offerId}`.
* Non-purchasable groups must not show active Buy actions.
* Invalid, missing, unavailable, stale, or missing-link offers must not appear as purchasable.
* The UI must not imply DealSach sells directly.
* The UI must not mention carts, checkout, payment, shipping, vouchers, coupons, reviews, or ratings.

### 6. Preserve or improve current search page integration

* Do not regress `SearchPage.tsx`.
* Search must remain API-backed through `GET /api/public/books`.
* Filters must remain API-backed through `GET /api/public/filters`.
* Search results must still show:

  * book cards
  * offer count, not retailer count
  * exactly one no-price status indicator where applicable
  * Vietnamese empty state and suggested next actions
  * pagination with default page size 12
* Search ranking must remain backend-driven.

### 7. Shared component cleanup

* Refactor `shared.tsx` only as needed to support the restored original visual system with API-backed data.
* Keep shared visual tokens centralized.
* Keep typed props for API-backed book cards and offer rows.
* Do not keep obsolete mock book arrays as active runtime catalog data.
* If mock arrays remain for temporary visual reference, isolate and label them clearly as non-runtime examples, but prefer removing them once API-backed equivalents exist.

### 8. Responsive and visual requirements

* Public pages must support 360px minimum width without unintended horizontal scrolling.
* Desktop layout must work at 1366px width.
* Verify Vietnamese text does not overflow:

  * homepage carousel/banner
  * book cards
  * price-drop badges
  * popular-clicked badges
  * search input
  * offer rows
  * Buy buttons
  * unavailable/stale/missing-link labels
  * disclaimer blocks
* Preserve Neubrutalist behavior from the original UI where practical:

  * hard black borders
  * hard black shadows
  * sharp cards
  * button depression on hover/focus where already used
  * visible keyboard focus

### 9. No backend or data-contract changes

* Do not change backend APIs.
* Do not change backend eligibility rules.
* Do not change Buy-flow persistence.
* Do not change database schema.
* Do not change seed data unless a blocker is documented and explicitly justified.
