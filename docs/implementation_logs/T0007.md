# T0007 — Backend Email Verification and Session Foundation

## Goal

Add the backend account-access foundation so users can request a one-time email verification code, verify it, receive a 7-day authenticated session, read their current account state, and log out through JSON APIs.

## Requirements

- Read `./AGENTS.md` before implementation.
- Add a new backend migration for account access tables.
- Add a `users` table with at least:
  - normalized unique email
  - display email
  - role, supporting `registered` and `admin`
  - status, supporting `active` and `deactivated`
  - alert email preference flag for future alert work
  - timestamps
- Add an `email_verification_codes` table with enough fields to enforce:
  - 10-minute expiry
  - 60-second resend cooldown
  - maximum 5 requests per email per hour
  - maximum 10 requests per email per day
  - maximum 5 failed attempts per active code
  - previous unused codes invalidated when a new code is requested
  - all active codes invalidated after successful verification
  - used codes cannot be reused
- Store verification codes as hashes, not plaintext.
- Add an `outbound_emails` or equivalent mock email outbox table for deterministic local/demo verification.
  - No external email provider is required.
  - Verification-code delivery must be recorded in this table.
  - Tests may inspect the outbox record to obtain the generated code.
  - The public request API must not return the verification code.
- Add a `user_sessions` table supporting:
  - multiple active sessions per user
  - opaque session token hash
  - 7-day expiry
  - logout invalidation
  - deactivated-user session rejection
- Add JSON auth endpoints:
  - `POST /api/auth/email-code/request`
  - `POST /api/auth/email-code/verify`
  - `GET /api/auth/me`
  - `POST /api/auth/logout`
- `POST /api/auth/email-code/request` must:
  - accept an email address
  - normalize email consistently
  - validate email format
  - return the same neutral Vietnamese success message whether the email is new or already registered
  - block deactivated accounts from receiving new login codes with a neutral or safe Vietnamese response that does not leak unnecessary account details
  - enforce cooldown and hourly/daily request limits
  - invalidate older unused active codes for the same normalized email
  - record a mock outbound email containing the verification code
- `POST /api/auth/email-code/verify` must:
  - accept email and code
  - validate the latest active code for that email
  - reject expired, used, invalidated, or over-attempted codes
  - increment failed attempts on wrong code
  - create a new active registered user when the email is new
  - log in an existing active user when the email already exists
  - block deactivated users from login
  - invalidate all other active codes for the email after success
  - mark the successful code as used
  - create a 7-day session
  - set an HTTP-only session cookie suitable for same-origin frontend use
- `GET /api/auth/me` must:
  - return guest state when no valid session is present
  - return the authenticated user’s id, email, role, status, and alert email preference when a valid session is present
  - reject expired sessions
  - reject sessions for deactivated users and invalidate those sessions
- `POST /api/auth/logout` must:
  - invalidate the current session if one exists
  - clear the session cookie
  - return a Vietnamese success response even if the user was already logged out
- API responses must follow the existing JSON shape:
  - `status`
  - `message`
  - `data`
  - `errors`
- User-facing messages and validation errors must be Vietnamese-first.
- Use `Asia/Ho_Chi_Minh` for expiry, cooldown, request-window, and session timestamps.
- Do not expose verification-code hashes, raw session tokens, or internal secret values through API responses.
- Do not implement wishlist APIs in this ticket.
- Do not implement price-alert APIs in this ticket.
- Do not implement Admin UI or Admin CRUD in this ticket.
- Do not add external email, queue, auth, JWT, or rate-limit dependencies.

## Acceptance Criteria

- [ ] New account-access migration creates `users`, `email_verification_codes`, `user_sessions`, and mock outbound-email storage.
- [ ] Migration rollback removes only the new account-access tables in safe dependency order.
- [ ] `php spark migrate` passes on a clean disposable Docker database.
- [ ] `POST /api/auth/email-code/request` validates email and returns Vietnamese JSON.
- [ ] Requesting a verification code records a mock outbound email and does not return the code in the API response.
- [ ] Verification-code expiry is 10 minutes.
- [ ] Verification-code resend cooldown is 60 seconds.
- [ ] Request limits are enforced at 5 per email per hour and 10 per email per day.
- [ ] Requesting a new code invalidates previous unused codes for the same email.
- [ ] A code allows no more than 5 failed attempts.
- [ ] Used, expired, invalidated, and over-attempted codes cannot be used.
- [ ] Successful verification creates a new active registered user for a new email.
- [ ] Successful verification logs in an existing active user without creating a duplicate user.
- [ ] Successful verification invalidates all other active codes for the same email.
- [ ] Successful verification creates a 7-day session and sets an HTTP-only session cookie.
- [ ] Multiple active sessions for the same user are supported.
- [ ] `GET /api/auth/me` returns guest state without a valid session.
- [ ] `GET /api/auth/me` returns current user state with a valid session.
- [ ] Expired sessions are rejected.
- [ ] Sessions for deactivated users are rejected and invalidated.
- [ ] `POST /api/auth/logout` invalidates the current session and clears the auth cookie.
- [ ] Auth endpoints use the existing JSON response shape.
- [ ] Public catalog routes still return HTTP 200 after the auth migration.
- [ ] Buy redirect behavior still works after the auth migration.
- [ ] Backend PHPUnit passes.
- [ ] `docs/Repo_Current_State.md` is updated with implementation and verification results.
- [ ] `docs/Known_Issues_And_Followups.md` is updated only if new issues are found.


```bash
docker compose -p dealsach_t0007 run --rm app sh -lc 'cd backend && php spark routes | grep -E "api/auth|email-code|logout|me"'
```

9. Request a verification code:

```bash
curl -i -c /tmp/dealsach-auth-cookies.txt \
  -H 'Content-Type: application/json' \
  -d '{"email":"tester@example.com"}' \
  http://localhost/api/auth/email-code/request
```

10. Expected result: JSON success response in Vietnamese; response body does not include the verification code.

11. Inspect the mock outbound email record from the database or a test helper query documented by Codex.

13. Verify the code:

```bash
curl -i -b /tmp/dealsach-auth-cookies.txt -c /tmp/dealsach-auth-cookies.txt \
  -H 'Content-Type: application/json' \
```
