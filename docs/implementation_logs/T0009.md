# T0009 — Price Alert Domain and Authenticated Management APIs

## Goal

Allow registered users to create, view, update, pause, reactivate, renew, restart, disable, and configure price alerts through authenticated JSON APIs without evaluating alerts or sending alert emails.

## Requirements

- Read `./AGENTS.md` before implementation.
- Add new backend migrations for alert persistence and account-level alert email preferences.
- Add a `price_alerts` table with at least:
  - `id`
  - `user_id`
  - `book_id`
  - `alert_type`
  - `status`
  - `target_price`
  - `baseline_price`
  - `baseline_pending`
  - `comparison_price`
  - `last_notified_price`
  - `notification_count`
  - `expires_at`
  - timestamps
  - referential safeguards for users and books.
- Supported alert types:
  - `target_price`
  - `new_lowest_price`
- Supported alert statuses:
  - `Active`
  - `Paused`
  - `Auto-paused`
  - `Expired`
  - `Disabled`
- Add a `price_alert_events` table with at least:
  - `id`
  - `price_alert_id`
  - `event_type`
  - `previous_status`
  - `new_status`
  - `summary_json`
  - `created_at`
- Add a `user_alert_preferences` table or equivalent normalized persistence with at least:
  - `user_id`
  - `alert_emails_enabled`
  - timestamps
- Default `alert_emails_enabled` must be enabled for users without an explicit preference row.
- Add models for alert, alert event, and alert preference persistence.
- Add `PriceAlertService` for all alert business logic.
- Add `AlertPreferenceService` for account-level alert email preference logic.
- Add authenticated JSON alert endpoints:
  - `GET /api/user/alerts`
  - `GET /api/user/alerts/{alertId}`
  - `POST /api/user/alerts`
  - `PATCH /api/user/alerts/{alertId}`
  - `POST /api/user/alerts/{alertId}/pause`
  - `POST /api/user/alerts/{alertId}/reactivate`
  - `POST /api/user/alerts/{alertId}/renew`
  - `POST /api/user/alerts/{alertId}/restart-tracking`
  - `POST /api/user/alerts/{alertId}/disable`
- Add authenticated JSON alert preference endpoints:
  - `GET /api/user/alert-preferences`
  - `PATCH /api/user/alert-preferences`
- All alert and alert-preference endpoints must require a valid active authenticated session.
- Guest requests must return Vietnamese JSON with HTTP 401 and the existing response shape:
  - `status`
  - `message`
  - `data`
  - `errors`
- Users must only access their own alerts.
- Accessing another user’s alert must return Vietnamese JSON without exposing whether the alert exists.
- Alert list/detail responses must include enough information for a later frontend management page:
  - alert id
  - book id
  - book title
  - author
  - publisher
  - category name
  - category slug
  - cover image
  - alert type
  - status
  - target price when applicable
  - baseline price when applicable
  - baseline pending flag
  - comparison price when applicable
  - last notified price
  - notification count
  - expiry timestamp
  - current lowest eligible price summary when available
  - account-level alert email enabled flag
  - recent alert events.
- Target-price alert creation must:
  - require an Active book.
  - require `target_price` as a Vietnamese Dong whole number greater than 0.
  - reject missing, zero, negative, decimal, or non-numeric target prices with Vietnamese validation JSON.
  - use current eligible-offer rules to determine the current lowest eligible price.
  - if a current eligible price exists and is at or below target, store that current price as the comparison price and send no email.
  - if a current eligible price exists and is above target, store the current price as the comparison price and send no email.
  - if no current eligible price exists, create the alert in a pending-comparison state and send no email.
  - set notification count to `0`.
  - set expiry to 90 days from creation using `Asia/Ho_Chi_Minh`.
- New-lowest-price alert creation must:
  - require an Active book.
  - not require or accept `target_price`.
  - use current eligible-offer rules to determine whether an eligible price exists at creation.
  - if a current eligible price exists, set baseline from the lowest observation-time eligible price known for that book at creation time.
  - if no current eligible price exists, set `baseline_pending` to true.
  - not send an immediate email.
  - set notification count to `0`.
  - set expiry to 90 days from creation using `Asia/Ho_Chi_Minh`.
- Duplicate prevention must follow AL-019 through AL-024:
  - Active, Paused, and Auto-paused alerts count as non-terminal existing alerts.
  - Expired and Disabled alerts do not block creating a new alert.
  - A user may have at most one non-terminal target-price alert for the same book and target price.
  - A user may have multiple target-price alerts for the same book when target prices differ.
  - A user may have at most one non-terminal new-lowest-price alert per book.
  - A user may have both target-price and new-lowest-price alerts for the same book.
  - Creating a duplicate non-terminal alert must return the existing alert instead of creating a duplicate row.
- Alert update behavior must follow allowed status permissions:
  - Active target-price alerts can update target price, pause, or disable.
  - Paused target-price alerts can update target price, reactivate, or disable.
  - Auto-paused alerts can reactivate or disable only.
  - Expired alerts can renew or be replaced by a new alert; no normal update.
  - Disabled alerts are read-only and cannot reactivate.
- Updating a target price must:
  - validate duplicate rules using the new target price.
  - reset notification count to `0`.
  - clear last notified price.
  - recompute comparison price from current eligible price.
  - not send an immediate email.
  - record an alert event.
- Pause must:
  - change Active alerts to Paused.
  - treat already Paused alerts as no-op success.
  - reject Disabled alerts with Vietnamese JSON.
  - record an alert event only when state changes.
- Reactivate must:
  - change Paused alerts to Active when the user and book are eligible.
  - change Auto-paused alerts to Active and reset notification count to `0`.
  - reject Disabled alerts.
  - reject alerts for archived books.
  - record an alert event only when state changes.
- Renew must:
  - reset expiry to 90 days from renewal.
  - allow Expired alerts to become Active when the user and book are eligible.
  - allow Active, Paused, and Auto-paused alerts to extend expiry without changing status unless required by the rules.
  - reject Disabled alerts.
  - record an alert event.
- Restart tracking must:
  - apply only to new-lowest-price alerts.
  - be allowed only when the alert is Active or Paused.
  - reset notification count to `0`.
  - clear last notified price.
  - set current eligible price as the new baseline when available.
  - set baseline pending when no current eligible price exists.
  - not send an immediate email.
  - record an alert event.
- Disable must:
  - set the alert status to Disabled.
  - preserve alert history.
  - treat already Disabled alerts as no-op success.
  - record an alert event only when state changes.
- Expiry behavior must be implemented without a scheduler:
  - Service reads/actions may normalize past-due Active, Paused, or Auto-paused alerts to Expired.
  - Disabled alerts do not need expiry processing.
  - Expired alerts remain visible.
- Account-level alert email preference behavior must:
  - allow the authenticated user to read `alert_emails_enabled`.
  - allow the authenticated user to set `alert_emails_enabled` true or false.
  - not change individual alert statuses.
  - not evaluate alerts.
  - not record Triggered events.
  - return Vietnamese JSON.
- Add backend database and feature tests for:
  - alert schema creation and rollback safety.
  - target-price alert creation with valid target price.
  - target-price validation failures.
  - target-price duplicate prevention.
  - multiple target prices for the same book.
  - new-lowest alert creation with baseline.
  - new-lowest alert pending baseline when no current eligible price exists.
  - new-lowest duplicate prevention.
  - coexistence of target-price and new-lowest alerts for the same book.
  - list/detail owner scoping.
  - guest rejection.
  - invalid/deactivated session rejection.
  - archived/nonexistent book creation rejection.
  - target-price update reset behavior.
  - pause/reactivate/renew/restart/disable behavior.
  - Disabled alerts cannot be reactivated.
  - Expired and Disabled alerts do not block creation of new alerts.
  - account-level alert email preference read/update behavior.
  - no outbound alert emails are created by create/update/actions.
  - public catalog and Buy-flow smoke stability after new migrations.
- Update `DealSachDemoSeeder` only as needed so reseeding clears alert and alert-preference tables safely before dependent account/catalog rows.
- User-facing messages and validation errors must be Vietnamese-first.
- Use `Asia/Ho_Chi_Minh` for alert timestamps, expiry, and time comparisons.
- Do not create Playwright, browser-driven, screenshot, or UI capture tests.
- Do not add new PHP or Node dependencies.
