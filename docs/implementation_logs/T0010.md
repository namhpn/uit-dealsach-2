## Requirements

* Add frontend TypeScript DTOs and API helpers for:

  * `GET /api/user/alerts`
  * `GET /api/user/alerts/{alertId}`
  * `POST /api/user/alerts`
  * `PATCH /api/user/alerts/{alertId}`
  * `POST /api/user/alerts/{alertId}/pause`
  * `POST /api/user/alerts/{alertId}/reactivate`
  * `POST /api/user/alerts/{alertId}/renew`
  * `POST /api/user/alerts/{alertId}/restart-tracking`
  * `POST /api/user/alerts/{alertId}/disable`
  * `GET /api/user/alert-preferences`
  * `PATCH /api/user/alert-preferences`
* Add an authenticated `/alerts` page that:

  * Opens the existing auth dialog for guests instead of duplicating auth UI.
  * Lists the current user’s alerts.
  * Shows book title/cover fallback/category, alert type, status, target price or baseline fields, expiry, notification count, and current/comparison price when returned by the API.
  * Provides valid lifecycle actions according to alert type/status.
  * Handles loading, empty, success, and Vietnamese API error states.
* Add alert creation from book detail:

  * Target-price alert form with VND integer validation.
  * New-lowest-price alert creation.
  * Guest users are prompted through the existing auth dialog.
  * Duplicate-create API success is treated as a normal success, not as a frontend error.
* Add account-level alert email preference controls on the alerts page:

  * Read default preference from the API.
  * Toggle `alert_emails_enabled`.
  * Explain that this preference suppresses alert emails without changing individual alert statuses.
* Keep the existing Neubrutalist public UI direction and reuse current shared constants/components where practical.
* Do not implement alert evaluation, scheduler behavior, email sending, or backend changes.
* Do not add new runtime dependencies.
* Do not create browser/UI automation tests or browser screenshots.

## Non-Goals
