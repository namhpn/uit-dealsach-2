## Requirements

* Add an Admin authorization layer:

  * Only authenticated users with `role: "admin"` may access Admin APIs.
  * Non-admin authenticated users receive a Vietnamese 403 JSON response.
  * Guests receive a Vietnamese 401 JSON response.
  * Covers AD-001, AD-004, AD-005, AD-006, NFR-024.
* Add deterministic first-admin setup support:

  * Provide a Spark command or seed-safe mechanism to promote/create the first Admin account for project setup.
  * Must not expose public self-service Admin escalation.
  * Covers AD-002 and AD-003 within project mock/setup scope.
* Add Admin audit persistence:

  * Audit records include Admin actor, action type, entity type, entity identifier, Vietnam-time timestamp, changed-field summary, and before/after values for non-sensitive fields.
  * Sensitive values are masked.
  * Read-only Admin views are not audited.
  * Covers AD-016 through AD-030 and SF-001 through SF-006.
* Add backend Admin APIs for:

  * Current Admin session/status.
  * User list with search/filter by email, role, and status.
  * User detail with wishlist/alert summary counts.
  * Deactivate user.
  * Reactivate user.
  * Alert activity list.
  * Alert detail/activity history.
  * Admin disable problematic alert.
  * Audit-log list.
* User deactivation behavior:

  * Prevent future login for deactivated users.
  * Invalidate active sessions for deactivated users.
  * Disable that user’s Active alerts.
  * Preserve historical wishlist, alert, email-link, click, and report records.
  * Audit success and failure paths.
  * Covers AV-020, AV-021, AV-030 through AV-036.
* Admin safety rules:

  * Last active Admin cannot be deactivated.
  * Admin Users cannot deactivate their own account.
  * Failed attempts for those rules are audited.
  * Covers AD-007, AD-008, AD-027, AD-028, AD-029, AD-030.
* Admin alert disable behavior:

  * Admin disable sets alert status to `Disabled`.
  * Stops future alert evaluation and future alert emails.
  * Preserves alert history.
  * Does not delete alerts.
  * Records both alert event and Admin audit event.
  * Covers AD-011 through AD-015.
* Add frontend Admin shell:

  * Route group such as `/admin`, `/admin/users`, `/admin/alerts`, `/admin/audit`.
  * Uses existing auth state.
  * Shows Vietnamese 401/403 states without duplicate auth logic.
  * Admin navigation is available only for Admin users.
  * Minimum Admin layout width may be 768px; no browser capture tests.
* Add frontend Admin Users page:

  * User list with email, role, status, alert-email preference, and basic counts when available.
  * Deactivate/reactivate actions with Vietnamese confirmations/messages.
  * Clearly prevents or explains self/last-admin deactivation failures from API messages.
* Add frontend Admin Alerts page:

  * Alert activity list with user email, book title, alert type, status, notification count, and recent event metadata.
  * Admin disable action for eligible problematic alerts.
* Add frontend Admin Audit page:

  * Read-only audit table with timestamp, actor, action, entity type/id, summary, and masked before/after data.
* Keep Vietnamese-first messages across backend and frontend.
* Add focused backend tests for Admin authorization, first-admin setup, user deactivation/reactivation, session invalidation, last-admin/self-deactivation protections, alert disabling, and audit masking.
* Add frontend build-safe Admin pages without browser/UI automation.
* Do not add new Composer or npm dependencies.

## Acceptance Criteria

* [ ] A setup command or deterministic seed-safe flow can create/promote the first Admin account.
* [ ] Admin-only APIs reject guests with 401 and non-admin users with 403 using Vietnamese JSON envelopes.
* [ ] Admin user list/detail APIs return expected user and summary data.
* [ ] Admin can deactivate a registered user.
* [ ] Deactivated users cannot log in and active sessions are invalidated.
* [ ] Active alerts owned by a deactivated user become `Disabled`.
* [ ] Admin can reactivate a deactivated user without automatically reactivating old alerts.
* [ ] Last active Admin deactivation is blocked and audited.
* [ ] Admin self-deactivation is blocked and audited.
* [ ] Admin can view alert activity and disable problematic alerts.
* [ ] Admin-disabled alerts stop future evaluation/email and preserve history.
* [ ] Admin audit records include actor, action, entity, timestamp, summary, and masked before/after data.
* [ ] Read-only Admin views do not create audit records.
* [ ] Frontend `/admin` shell, users, alerts, and audit pages compile and route correctly.
* [ ] Admin navigation is visible only to Admin users.
* [ ] Backend tests cover Admin authorization, user state transitions, alert disabling, and audit trail behavior.
* [ ] Frontend build passes.
* [ ] No browser automation, screenshots, or visual-regression artifacts are added.
* [ ] Documentation and repo state are updated.
