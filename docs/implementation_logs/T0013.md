## Requirements

* Add Admin-only backend APIs and frontend pages for:

  * Categories
  * Books
  * Retailer platforms
  * Merchants
  * Offers under books
  * Mock price observations under offers
* Reuse the T0012 Admin authorization mechanism:

  * Guests receive Vietnamese 401 JSON.
  * Non-admin users receive Vietnamese 403 JSON.
  * Admin users may access catalog management APIs.
* Reuse T0012 Admin audit logging for every mutation:

  * Create, update, archive, restore, and status changes.
  * Audit records must include Admin actor, action type, entity type/id, timestamp, summary, and non-sensitive before/after values.
  * Sensitive affiliate tracking parameters and full affiliate URLs must be masked in audit records.
  * Read-only Admin catalog views must not create audit records.
  * Covers AD-016 through AD-026 and SF-004 through SF-006.
* Category management:

  * List/search/filter categories.
  * Create category with non-empty name and slug.
  * Update category name/slug/display metadata.
  * Archive and restore category.
  * Archived categories are hidden from public filters.
  * Creating/updating books must require an Active category.
  * Covers Admin management area for Categories and input validation for Category name.
* Book management:

  * List/search/filter books by title, author, ISBN, category, status, and featured state.
  * Create book with required title and author.
  * Update book metadata: title, author, publisher, ISBN, cover image path, category, description/technical fields if already present, status, and featured flag.
  * Archive and restore books.
  * Archived books are hidden publicly.
  * Archived books cannot be newly wishlisted.
  * Existing wishlist entries remain visible with archived marker.
  * Active alerts for archived books become Paused; other alert statuses remain unchanged.
  * Restored books become public again when their category is Active; related alerts remain Paused until user reactivates them.
  * Covers AD management for Books, BR-016, and entity lifecycle rules.
* Retailer platform management:

  * List/search/filter retailer platforms.
  * Create retailer with non-empty name.
  * Update name/status and approved destination domains.
  * Archive and restore retailer platforms.
  * Archived retailers stop public display and alert evaluation for related offers.
  * Restoring a retailer does not automatically make related merchants/offers eligible unless their own statuses are valid.
  * Covers Admin management area for Retailer Platforms.
* Merchant management:

  * List/search/filter merchants.
  * Create merchant under a retailer platform.
  * Update merchant name, retailer association, and status.
  * Validate that merchant belongs to the selected retailer platform for offers.
  * Archive and restore merchants.
  * Archived merchants stop public display and alert evaluation for related offers.
  * Restoring a merchant does not automatically make related offers eligible unless offer and retailer status are valid.
  * Covers Admin management area for Merchants and merchant-retailer consistency validation.
* Offer management:

  * List offers globally and under a book.
  * Create offer under a book with retailer platform, merchant, status, and affiliate destination fields.
  * Update offer fields and status.
  * Validate merchant-retailer consistency.
  * Validate affiliate destination uses `https://` and matches approved domain rules.
  * Support statuses: Pending Review, Active / Purchasable, Unavailable, Inactive, Removed / Invalid.
  * Surface eligibility review information: missing destination, invalid destination, stale latest observation, unavailable latest observation, merchant/retailer mismatch, archived related entity, and current purchasability.
  * Do not hard-delete offers.
  * Covers Admin management area for Offers, EO-001 through EO-009 review visibility, and input validation for Affiliate destination.
* Mock price observation management:

  * Admin can view latest and historical observations for an offer.
  * Admin can add a mock price observation to an existing observation cycle or a new dated cycle.
  * Observations must store observation-time state needed for historical charts and alert evaluation consistency.
  * Observation availability and listed item price validation must follow existing price rules.
  * Admin pages may show exact last checked time.
  * Public pages must still not show exact observation timestamps.
  * Covers PD-014 through PD-028 and PD-041.
* Public API consistency:

  * Public search, filters, discovery, book detail, price history, and Buy-flow behavior must reflect Admin catalog changes through existing public APIs.
  * Public pages must still exclude archived books and invalid/ineligible offers.
  * Stale/unavailable/missing-link display behavior must remain consistent.
* Frontend Admin catalog UI:

  * Add Admin navigation entries for catalog management.
  * Provide list/detail/edit/create/archive/restore flows for categories, books, retailers, merchants, and offers.
  * Provide offer observation list and add-observation form.
  * Display Vietnamese success/error messages.
  * Handle loading, empty, validation-error, and unauthorized states.
  * Keep Admin pages usable at 768px minimum width.
* Tests:

  * Backend tests must cover Admin authorization, validation, catalog lifecycle effects, audit logging, public API effects, offer eligibility review, and observation-time state capture.
  * Frontend must compile with the new Admin catalog pages.
* Do not add new Composer or npm dependencies.
