## Requirements

* Create or replace the top-level `README.md`.
* README must include a clear project introduction:

  * Explain that DealSach is a Vietnamese-first book price tracker and affiliate deal platform.
  * Explain the problem it solves: book discovery, price comparison across tracked Vietnamese retailers/merchants, reference price history, wishlist, price alerts, affiliate redirects, Admin management, and dashboard reporting.
  * Clearly state DealSach does not sell books directly and does not provide checkout, payment, shipping, carts, reviews, or real-time price guarantees.
* README must include feature overview sections for:

  * Public homepage/search/book detail.
  * Price comparison and historical price observations.
  * Affiliate Buy redirect flow.
  * Email-code authentication.
  * Wishlist.
  * Price alerts and mock alert emails.
  * Email deal-link click tracking.
  * Account settings.
  * Admin users/alerts/audit.
  * Admin catalog management.
  * Admin dashboard reports.
  * Mock/demo data.
* README must include setup and installation:

  * Prerequisites.
  * Clone/open repository steps.
  * Environment file expectations using the existing repo state.
  * Dockerized dependency installation if applicable.
  * Docker Compose startup.
  * Backend migration and seeding.
  * Frontend build and dev server commands.
  * Backend PHPUnit command.
  * Frontend build command.
  * Clean disposable verification workflow.
* README must document known local setup caveats:

  * phpMyAdmin host port `8080` conflict from KI-0011 and a safe workaround or pointer to known issues.
  * Demo cover fallback behavior from KI-0009.
  * Dashboard seed-data caveat from KI-0013.
  * Category display metadata caveat from KI-0012.
* README must include usage examples:

  * Public browsing/search.
  * Opening a book detail page.
  * Clicking Buy through DealSach redirect.
  * Requesting/verifying email code in the UI or via documented API/curl examples.
  * Managing wishlist.
  * Creating/managing alerts.
  * Running `php spark alerts:evaluate`.
  * Using generated mock alert email links and disable links at a high level.
  * Signing in as seeded Admin.
  * Admin catalog management examples.
  * Admin dashboard/report examples.
* README must include API route overview:

  * Public catalog routes.
  * Auth routes.
  * Wishlist routes.
  * Alert routes.
  * Alert email-link routes.
  * Admin user/alert/audit routes.
  * Admin catalog routes.
  * Admin dashboard/report routes.
* README must include project structure overview:

  * `backend/`
  * `frontend/`
  * `docs/`
  * Docker/config files.
* README must include testing and verification section:

  * Backend tests.
  * Frontend build.
  * PHP lint command.
  * Route inspection examples.
  * Manual verification guide pointer.
* README must include development workflow guidance:

  * Ticket-driven implementation process.
  * Completion report expectations.
  * No browser automation/screenshots unless explicitly allowed by a future ticket.
  * Do not add dependencies casually.
* README must use concise, accurate wording and avoid claiming unsupported production readiness.
* README must keep the project language context clear:

  * Document in English.
  * Note that product UI/messages are Vietnamese-first.
* README must not include secrets, real credentials, private tokens, or copied `.env` contents.
* README must not introduce new commands that conflict with current docs; commands should be based on existing repo-standard Docker/manual verification commands.
* Update `docs/Manual_Verification_Guide.md` only if adding a README verification section or pointer improves consistency.
* Update `docs/Repo_Current_State.md` with T0015 completion state once implemented.
* Update `docs/Known_Issues_And_Followups.md` only if the README work reveals documentation gaps outside scope.
* Add `docs/implementation_logs/T0015.md` with ticket text and completion report.

   ```bash
   git diff --check
   ```


