# T0017 — Add Design Documentation and README Architecture

## Goal

Add the missing design documentation deliverables and refactor README so the project clearly documents database design, UML analysis, and system architecture.

## Dependencies

* T0016 completed implementation state.
* `docs/requirement-doc.md`
* `docs/Repo_Current_State.md`
* Existing backend migrations under `backend/app/Database/Migrations/`
* Existing frontend and backend source organization.
* Existing README.

## Allowed Areas

Codex may edit only:

* `docs/design/ERD.md`
* `docs/uml/`
* `README.md`

Always allowed for ticket bookkeeping:

* `docs/Repo_Current_State.md`
* `docs/Known_Issues_And_Followups.md`

Codex must not edit:

* Backend source code.
* Frontend source code.

* Runtime behavior.
* `docs/requirement-doc.md`

## Requirements

* Create `docs/design/ERD.md`.
* Include a Mermaid `erDiagram` block for the current DealSach database.
* Cover at least:

  * Catalog tables.
  * Buy-flow tables.
  * Account/session/email tables.
  * Wishlist tables.
  * Price-alert tables.
  * Alert notification/link tracking tables.
  * Admin audit tables.
* Add concise ERD notes explaining key design decisions:

  * One primary category per book.
  * Offers connect books, retailer platforms, and merchants.
  * Price observations preserve observation-time facts.
  * Historical/action records are retained instead of hard-deleted.
* Create markdown-only UML documentation under `docs/uml/`.
* Do not generate diagram image files.
* UML files must describe proposed diagrams and their participants/flows in text only.
* Include at least:

  * Use-case documentation for Guest User, Registered User, and Admin User.
  * Activity-flow documentation for discovery/buy, email verification, wishlist, price alert, and admin catalog management.
  * Sequence-flow documentation for public search, Buy redirect, email-code login, alert evaluation, and admin mutation/audit.
  * Component/deployment documentation describing React frontend, Nginx, CodeIgniter backend, MariaDB, SMTP provider, and browser/client boundary.
* Refactor README:

  * Remove unnecessary detailed “Testing and Verification”, “Known Local Caveats”, and “Deeper Documentation” sections from README.
  * Add a direct “System Architecture” section to README.
  * Keep README concise and useful as the project entry point.
  * Preserve setup/install basics.
  * Preserve core feature summary.
  * Point to `docs/design/ERD.md` and `docs/uml/` where appropriate.
* Keep documentation language clear and aligned with the current implementation.
* Do not claim any behavior that is not implemented.
