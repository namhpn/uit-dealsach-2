## Requirements

### Product direction

- ProductDetailPage must be primarily a **price comparison decision page**.
- Book information supports the decision, but the primary hierarchy must be:
  1. book identity,
  2. best current eligible price,
  3. seller comparison,
  4. price alert,
  5. trust/disclaimer,
  6. supporting book details.
- Use commerce-first Neubrutalism:
  - sharp corners,
  - black borders,
  - hard shadows,
  - flat colors,
  - Be Vietnam Pro,
  - no gradients,
  - no soft shadows,
  - no rounded cards/buttons/inputs.

### Hero section

- Keep a visually quiet breadcrumb.
- Keep the current description as a text/blockquote-style description; do **not** convert it into a bordered description card.
- Make the single cover image visually strong and large.
- Do not implement real carousel behavior.
- Remove or simplify disabled carousel controls if they add misleading affordance.
- Use `contain` for real cover images so book covers are not cropped.
- Use a structured fallback background/frame around cover images.
- Keep a small alert CTA near the best-price panel, but do not put the full alert form in the hero.

### Pricing model

- Best price must use only current purchasable eligible offers.
- Stale prices must not appear as the main best price.
- If no eligible current price exists, show the public status label.
- Add ProductDetailPage support for highest/current reference price using current eligible offers:
  - Extend the public book detail summary with `highest_eligible_price`.
  - Frontend type must include this field.
  - Show a strikethrough/reference price only when:
    - `lowest_eligible_price` is not null,
    - `highest_eligible_price` is not null,
    - `highest_eligible_price > lowest_eligible_price`.
- Do not invent MSRP.
- Do not add discount percentage.
- If price-drop language is shown, derive it from existing price observation history and label it as `Giảm gần đây`.

### Seller comparison

- Keep grouped offers:
  - purchasable,
  - unavailable,
  - stale reference,
  - missing valid seller link.
- Purchasable offers must appear first.
- Unavailable/stale/missing-link offers must remain visible below purchasable offers as reference rows.
- Do not use opacity or dashed borders as the primary unavailable-state styling.
- Use hard black borders, flat muted fills, and stamped status labels.
- Missing valid seller-link rows must show a hard-bordered disabled block labeled `CHƯA CÓ LIÊN KẾT`.
- Stale rows must show a hard-bordered disabled block labeled `GIÁ THAM KHẢO CŨ`.
- Unavailable rows must show a hard-bordered disabled block labeled `TẠM HẾT HÀNG`.
- Seller rows must expose merchant name as secondary text under retailer/platform.
- The primary CTA label must be `ĐẾN NƠI BÁN`.
- The cheapest/best offer CTA should use stronger Level 2 emphasis (`shadow8`), while secondary seller CTAs may use `shadow4`.

### Trust and disclosure

- Do not expose exact `latest_observed_at` publicly.
- Do not add affiliate disclosure near every CTA.
- Keep a full disclaimer/trust block at the bottom.
- Avoid repeating the price disclaimer under every seller row.
- Price disclaimer may appear once near the best price if needed, and must appear in the bottom disclaimer block.
- Bottom disclaimer must explain:
  - prices are references,
  - actual seller price may change,
  - DealSach does not sell directly,
  - DealSach does not handle payment, delivery, returns, or order support.

### Alerts

- Keep both alert types:
  - target price,
  - new lowest price.
- Move the full alert form into or directly below the price-history section.
- Unauthenticated users may see the alert form; submitting must open the auth dialog.
- Use the user-facing CTA text `THEO DÕI GIẢM GIÁ`.
- Inputs must use `3px` black border and visible focus styling.
- Disabled/busy buttons must retain hard structural borders.

### Price history chart

- Remove the area gradient.
- Use flat fill or line-only chart.
- Tooltip must remain square-cornered with black border.
- Do not add new chart dependencies.

### Technical details metadata

- Check current schema first.
- If fields do not already exist, add these book metadata fields:
  - `release_date` nullable date,
  - `page_count` nullable unsigned integer,
  - `dimensions` nullable string,
  - `format` nullable string.
- Do not add weight, language, edition, or book gallery fields in this ticket.
- Public book detail response must expose the new metadata fields under `book`.
- Admin book APIs must create/update/read the new fields.
- Admin UI must allow detail editing or row-level editing of the new metadata fields.
- Admin quick-create may remain minimal.
- Seeded demo books must include representative values for the new metadata fields.
- Missing metadata must render as `Chưa cập nhật`.

### Related books

- Keep the existing frontend discovery fallback for related books.
- Show at most 4 related books.
- Ensure the related-books layout fits desktop columns cleanly.
- Do not add backend `related_books`, `relevant_books`, or related-ranking logic in this ticket.

### Accessibility and responsive behavior

- Preserve keyboard focus visibility.
- Preserve wishlist auth-dialog behavior.
- Preserve price-alert auth-dialog behavior.
- Seller CTAs must remain keyboard-accessible.
- Mobile layout must stack cleanly:
  1. cover,
  2. title/metadata,
  3. best price,
  4. small alert/save actions,
  5. seller comparison,
  6. price history + alert form,
  7. technical details,
  8. related books,
  9. disclaimer.
- Avoid horizontal overflow on 360px mobile width.
