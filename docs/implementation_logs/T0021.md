## Requirements

### 1. Preserve the correct production base

- Keep `/search` routed to `frontend/src/app/pages/SearchPage.tsx`.
- Do not copy `frontend-original/src/app/pages/SearchResultPage.tsx` wholesale.
- Adapt the styling, layout patterns, and interaction language from `frontend-original/src/app/pages/SearchResultPage.tsx`.
- Preserve API-backed behavior through `GET /api/public/books` and `GET /api/public/filters`.
- Preserve backend-driven search ranking, filtering, sorting, and pagination.
- Remove any mock search corpus, mock filter arrays, local sorting, local filtering, or local pagination patterns if introduced during implementation.

### 2. Visual direction

- SearchPage must become a SearchResultPage-quality experience using DealSach commerce-first Neubrutalism:
  - sharp corners,
  - black borders,
  - hard shadows,
  - flat colors,
  - Be Vietnam Pro,
  - no gradients,
  - no soft shadows,
  - no rounded cards/buttons/inputs.
- Use the original SearchResultPage as the primary visual reference for:
  - filter/sidebar block,
  - emerald result hero,
  - query chip,
  - active filter chips,
  - hard-shadow pagination,
  - compact product grid,
  - bottom disclaimer.
- Do not copy original mock-data behavior.
- Keep the page optimized for book discovery first and deal cues second.

### 3. Search result hero

- Add a hard-bordered emerald result hero adapted from `frontend-original`.
- Hero copy must follow this hierarchy:

  ```text
  Kết quả tìm kiếm cho <search query>
  Tìm thấy {total} đầu sách
```

* If no search query exists, use a neutral heading such as:

  ```text
  Kết quả tìm kiếm
  ```

* Show the active search query as a hard-bordered chip when `q` exists.

* Do not show retailer count or seller count in the hero unless the public API returns query-specific retailer count.

* Hero must include or visually pair with the sort control.

* Hero must not imply DealSach sells books directly.

### 4. Filter model

* Check current public search API support before changing filter behavior.
* Implement all feasible filter enhancements against the current API.
* Remove the `Tình trạng` / availability filter from the visible SearchResultPage UI.
* Keep unavailable, stale-reference, missing-link, and no-tracked-offer books visible in results.
* Keep category filtering.
* Keep author filtering.
* Keep publisher filtering.
* Keep retailer filtering if the current API supports it.
* Keep free-text numeric price inputs:

  * `min_price`
  * `max_price`
* Use native inputs/selects styled as Neubrutalist controls.
* Inputs must use `3px` black border and visible focus styling.
* If the current API supports only single-value filters, keep visible filters single-select.
* If implementing multi-select support, update backend validation, filtering, frontend query serialization, tests, and manual verification in the same ticket.
* Do not implement hidden or fake multi-select UI that the backend cannot satisfy.

### 5. Active filter chips

* Add active filter chips for visible active filters:

  * query,
  * category,
  * author,
  * publisher,
  * retailer,
  * min price,
  * max price,
  * sort if non-default.
* Chips must be hard-bordered, sharp, removable buttons.
* Removing a chip must update the URL query parameters and reset `page=1`.
* Add a `Xóa bộ lọc` action when any removable filter is active.
* Do not render a chip for the removed availability filter.

### 6. Sorting

* Check current public search API sort support before rendering options.
* Render only sort options accepted by the backend and returned by `/api/public/filters`.
* Keep default sort as backend relevance/default behavior.
* Keep `Mới thêm gần đây` only if currently supported by API.
* Add `Giảm giá nhiều nhất` only if backend support is implemented in this ticket.
* If adding a discount/drop sort:

  * add backend validation,
  * add backend sort behavior,
  * expose the sort option from `/api/public/filters`,
  * add PublicCatalogApiTest coverage.

### 7. Search cards

* Use current `ApiBookCard` as the base search card.
* Preserve semantic link navigation to `/book/{bookId}`.
* Preserve wishlist behavior and auth-dialog behavior.
* Preserve compact price disclaimer behavior.
* The card must show:

  * cover or fallback,
  * category,
  * title,
  * author,
  * current lowest eligible price or one compact public status label,
  * offer count.
* Change the offer-count label from `ưu đãi` to `NƠI BÁN` on search result cards if needed by extracting a search-specific display prop or search-specific card variant.
* Do not show duplicate no-price status labels.
* Do not show direct external seller links on search cards.
* Card click target must remain the book detail route, not `/go/offers/{offerId}`.

### 8. Reference/strikethrough price on search cards

* Check current public search-card DTO support first.
* If `highest_eligible_price` is already available on public book card DTOs, use it.
* If it is absent, extend `GET /api/public/books` card DTOs to include `highest_eligible_price`, derived only from current purchasable eligible offers.
* Show a strikethrough/reference price only when:

  * `lowest_eligible_price` is not null,
  * `highest_eligible_price` is not null,
  * `highest_eligible_price > lowest_eligible_price`.
* Do not invent MSRP.
* Do not add a database `original_price` or MSRP field.
* Do not show discount percentage.

### 9. Price-drop badges and deal cues

* Check current public search API support first.
* If `price_drop` is already available on `GET /api/public/books`, render a compact price-drop badge on search cards.
* If `price_drop` is absent, add public search-card `price_drop` support using existing price-observation data.
* Price-drop metadata must be derived from real eligible price observations, not invented from reference price.
* Price-drop badge must use a hard-bordered red/deal-accent block and must be visually secondary to title and current price.
* If no real price-drop metadata exists for a result, do not render a badge.
* Do not add discount percentages.

### 10. Facets and counts

* Check current API before implementation.
* If the current API does not support faceted counts, do not fake counts in the UI.
* Optional: if backend support can be implemented cleanly within this ticket, add query-aware facet counts for visible filters only.
* If facet counts are added:

  * expose them from `/api/public/filters` or `/api/public/books` in a documented DTO shape,
  * ensure counts respect current query context,
  * add PublicCatalogApiTest coverage.
* Do not block the visual refresh on facet counts.

### 11. Pagination

* Keep backend default page size of 12.
* Do not add a page-size selector.
* Replace the simple previous/next controls with numbered Neubrutalist pagination.
* Pagination must use backend `pagination.page`, `pagination.total_pages`, and URL `page`.
* Buttons must be square, hard-bordered, and keyboard-accessible.
* Current page may use depressed/active Neubrutalist styling.
* Disabled buttons must keep structural borders.

### 12. Empty, loading, and error states

* Loading state must be hard-bordered and visually consistent.
* Empty state must use backend `empty_state.message` when available.
* Empty state must add actions:

  * `Xóa bộ lọc`
  * `Về trang chủ`
* Add frontend validation for price range:

  * non-negative integer VND,
  * `min_price <= max_price`.
* If backend returns validation errors, surface the most specific useful Vietnamese message available.
* If API error typing must be improved in `api.ts`, keep the change minimal and compatible with existing callers.

### 13. Accessibility

* Use real native form controls for filters.
* Mobile filter toggle must expose `aria-expanded`.
* Collapsible filter groups, if implemented, must expose useful `aria-expanded` semantics.
* Active filter chips must be buttons.
* Product cards must remain keyboard-focusable links.
* Wishlist controls inside cards must remain accessible and must not trigger card navigation.
* All interactive controls must have visible keyboard focus.
* Mobile layout must work at 360px without unintended horizontal scrolling.

### 14. Responsive layout

* Desktop:

  * left filter sidebar,
  * right result column,
  * result grid should fit cleanly with sidebar.
* Tablet:

  * filter panel may remain inline or collapsible if layout stays readable.
* Mobile:

  * filter toggle,
  * stacked hero,
  * one-column result grid,
  * no horizontal overflow from hard shadows.
* Result grid target:

  * 1 column mobile,
  * 2 columns small/tablet,
  * 3 columns when sidebar plus content width requires it,
  * 4 columns on wide desktop if cards remain readable.

### 15. Category archive consistency

* Check current backend behavior for active books joined to archived categories.
* If current search can return active books whose category is archived while the category is absent from filters, fix the public catalog query to exclude archived categories or document the intended behavior.
* Add PublicCatalogApiTest coverage if behavior changes.
* Keep this fix scoped to public catalog search/filter consistency only.

### 16. Tests and documentation

* Add or update PublicCatalogApiTest coverage for any API/DTO/filter/sort changes made in this ticket.
* Update frontend types in `frontend/src/app/api.ts` for new public search-card fields.
* Update manual verification guide with SearchResultPage-specific steps.
* Create `docs/implementation_logs/T0021.md`.
* Update `docs/Repo_Current_State.md`.
* Add or update known issues only for out-of-scope findings.
