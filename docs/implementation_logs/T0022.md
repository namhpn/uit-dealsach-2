# T0022 — Homepage Visual Rhythm Refinement

## Goal

Refine the current API-backed `frontend/` Homepage visual presentation by selectively restoring the stronger `frontend-original/` homepage merchandising rhythm while preserving the current production data model, copy, CTA behavior, wishlist behavior, and public catalog integration.

## Dependencies

* T0021 completed and reviewed.
* T0019 completed Homepage refresh.
* Existing current Homepage:

  * `frontend/src/app/pages/HomePage.tsx`
  * `frontend/src/app/shared.tsx`
  * `frontend/src/app/api.ts`
* Existing visual reference:

  * `frontend-original/src/app/pages/HomePage.tsx`
  * `frontend-original/src/app/shared.tsx`
* Existing public discovery API:

  * `GET /api/public/discovery`
* Existing public filters API:

  * `GET /api/public/filters`
* Existing wishlist/auth behavior on homepage cards.
* Current search-card/shared-component behavior from T0021 must not regress.

## Requirements

### 1. Preserve production Homepage foundation

* Keep the current `frontend/` Homepage API-backed.
* Continue loading homepage data from existing `fetchDiscovery()` and `fetchFilters()`.
* Do not reintroduce mock homepage data from `frontend-original/`.
* Preserve current `frontend/` hero copy.
* Preserve current `frontend/` hero CTA actions:

  * search action,
  * route action,
  * anchor scroll action.
* Do not add backend/admin homepage banner management.
* Do not add backend API fields.

### 2. Hero banner carousel

Implement a visually stronger hero while preserving current production semantics.

* Keep current `frontend/` banner copy and CTA actions.
* Reference `frontend-original/` hero for:

  * stronger campaign-like presence,
  * stronger image/card composition,
  * stronger spacing rhythm,
  * stronger CTA visual weight.
* Do **not** implement the original background graphics.
* Do **not** add circles, decorative geometric outlines, or new background ornaments.
* Keep the hero sharp-edged:

  * flat background,
  * black border,
  * hard shadow,
  * no gradient,
  * no rounded corners.
* Improve hero visual punch through typography, spacing, image framing, and CTA treatment only.
* Keep mobile layout readable and avoid horizontal overflow.

### 3. Featured category section

Refactor the featured category area to borrow the original interaction model.

* Use an emerald header bar.
* Use a left category rail.
* Show one active category at a time.
* Show one carousel for the active category.
* Make the category area feel like navigation, not a static explanation panel.
* Use current API-backed featured books grouped by category.
* Use category `display_label` with fallback to category `name`.
* Use category `display_description` where helpful, but do not let description copy dominate the active rail.
* Preserve `Xem tất cả` / category navigation links where already supported.
* Do not add new category API fields.
* Do not add per-category featured ordering.
* Do not add homepage admin controls.

### 4. Compact book cards in featured carousel

Update compact homepage book cards to match the intended e-commerce pattern.

Cards must show:

* cover or fallback;
* wishlist button;
* category;
* title;
* author;
* lowest eligible price or status label;
* original/reference price immediately below the current price when available;
* seller count label using `Nơi bán`, not `ưu đãi`;
* no CTA text such as `Đến nơi bán`.

Behavior:

* The entire card remains clickable.
* Card click navigates to `/book/{bookId}`.
* Wishlist button must not trigger card navigation.
* Wishlist button must preserve auth-dialog behavior for unauthenticated users.
* Reference/strikethrough price must only appear when existing `highest_eligible_price > lowest_eligible_price`.
* Do not invent MSRP or discount percentage.
* Do not show individual price disclaimers on compact homepage cards.

Implementation constraint:

* If `ApiBookCard` is shared with SearchPage, avoid changing search behavior unintentionally.
* Prefer adding homepage-specific props or a homepage-specific variant over globally changing all card usages.
* SearchPage card requirements from T0021 must remain intact.

### 5. Deal book cards

Keep the current deal card data behavior but restore stronger original-style commercial force.

* Keep current API-backed card data.
* Keep current card click/navigation behavior to `/book/{bookId}`.
* Keep wishlist button behavior.
* Keep price-drop and popular-deal badges when present.
* Remove publisher from homepage deal cards.
* Increase CTA text prominence.
* Make the CTA visually more dominant, closer to `frontend-original/` deal card.
* Keep CTA text as `Đến nơi bán →`.
* CTA must still navigate to `/book/{bookId}`.
* Do not direct cards to `/go/offers/{offerId}`.
* Do not add `best_buy_action`.
* Preserve reference price display when existing data supports it.
* Do not add discount percentage.

Implementation constraint:

* If `ApiDealBookCard` is shared outside Homepage, avoid unintended visual changes elsewhere.
* Prefer a prop or homepage-specific variant if necessary.

### 6. Price disclaimer treatment

* Remove the loud standalone Homepage disclaimer block.
* Do not render individual card-level price disclaimers on Homepage cards.
* Do not remove ProductDetailPage disclaimer behavior.
* Do not remove backend `price_disclaimer` data.
* Do not remove disclaimer support from shared components if other pages use it.

### 7. Deal section headers

Update homepage deal section headers to follow the original visual rhythm.

Header must include:

* icon;
* uppercase title;
* bottom black rule;
* small `Xem thêm` button.

Remove from homepage deal section headers:

* subtitle;
* window label;
* paragraph-like explanatory copy.

Typography:

* Title should visually follow `frontend-original/` deal section title treatment:

  * uppercase,
  * extra-bold,
  * dominant,
  * tight tracking if already used in the original pattern.

Behavior:

* `Xem thêm` must link to existing `ctaHref` when available.
* Do not add new search sorts.
* Do not add new backend routes.

### 8. Responsive behavior

* Desktop:

  * hero image and text feel balanced;
  * featured section uses left category rail and right carousel;
  * deal sections retain horizontal carousel behavior.
* Tablet:

  * featured rail remains readable or stacks cleanly if width is constrained.
* Mobile:

  * no unintended horizontal overflow;
  * hero remains readable;
  * category rail must not create unusable narrow columns;
  * if necessary, category rail can become a horizontal nav strip while still preserving “one active category at a time.”
* Hard shadows must not push content outside viewport.

### 9. Preserve T0021/shared behavior

Because `frontend/src/app/shared.tsx` is shared across Homepage and SearchPage:

* Do not regress SearchPage active filter chips, cards, pagination, or price-drop badges.
* Do not change SearchPage offer-count label behavior.
* Do not remove SearchPage compact price disclaimer behavior if it currently uses it.
* Do not change SearchPage visible filters or sorting.
* Do not alter public catalog API types unless build failure requires a small compatibility fix.
