## Requirements

### 1. Preserve production SearchPage foundation

- Keep SearchPage API-backed.
- Continue loading result data through existing `fetchBooks(params)`.
- Continue loading filter metadata through existing `fetchFilters()`.
- Preserve URL-driven query/filter/sort/pagination behavior.
- Preserve existing wishlist behavior in search cards.
- Preserve card navigation to `/book/{bookId}`.
- Do not reintroduce mock books, mock filters, local mock sorting, or local mock pagination from `frontend-original/`.
- Do not add browser, Playwright, Puppeteer, screenshot, or visual-capture testing.

### 2. Filter sidebar container

- Make the filter panel feel lighter than the current `frontend/` version.
- Reduce filter-panel shadow weight.
- Prefer the lighter `frontend-original/` treatment:
  - white background,
  - black border,
  - smaller hard shadow,
  - cleaner header,
  - less heavy container presence.
- The panel must still read as a hard-edged Neubrutalist component.
- Do not make the panel soft, rounded, gradient-based, or blur-shadowed.
- Keep mobile filter toggle behavior and accessibility.

### 3. Filter controls

- Restyle the filter controls toward `frontend-original/`:
  - collapsible filter groups,
  - group separators,
  - compact group headings,
  - custom tactile checkbox-style rows where appropriate,
  - stronger interaction rhythm.
- Use real accessible form controls underneath any custom visual treatment where practical.
- The controls should feel more interactive and tactile than the current stacked select-heavy panel.
- Preserve current API-compatible filter behavior.
- Do not implement unsupported multi-select backend behavior unless already supported by current SearchPage/API.
- If a visual checkbox row is used for a single-select filter, ensure selecting it produces the same one-value URL parameter behavior as the current API supports.
- Keep price inputs as free-text numeric VND inputs.
- Keep frontend price-range validation.

### 4. Removed availability/status filter

- Keep the visible `Tình trạng` / availability filter removed.
- Do not reintroduce availability/status as a visible filter group.
- Keep unavailable, stale-reference, missing-link, and no-tracked-offer books visible in search results.
- Do not hide non-purchasable books at the page level.

### 5. Result hero

- Restyle the result hero closer to `frontend-original/`.
- Reduce the current overly broad vertical spacing.
- Make the hero a tighter branded module:
  - deep emerald background,
  - black border,
  - hard 8px shadow,
  - compact query label,
  - query chip,
  - result count,
  - integrated sort control.
- The hero must feel bold and branded, not sparse.
- Keep copy aligned with current SearchPage intent:
  - `Kết quả tìm kiếm cho`
  - query chip when a query exists,
  - `Tìm thấy {total} đầu sách`.
- Do not add fake retailer/seller count.
- Do not imply DealSach sells books directly.

### 6. Sort control

- Restyle sort control to follow `frontend-original/`:
  - compact label,
  - custom hard-shadow wrapper,
  - custom chevron icon,
  - square select,
  - visually integrated with the hero.
- Increase or preserve font size enough for readability.
- Do not make the sort control feel like a default browser select.
- Render only currently supported sort options from existing filters data.
- Preserve current sort URL behavior and page reset behavior.

### 7. Active filter chips

- Keep active filter chips.
- Remove the duplicate `Xóa bộ lọc` button that appears next to active filter chips.
- Keep only the `Xóa bộ lọc` action in the filter sidebar/header controls.
- Each individual chip must remain removable.
- Removing a chip must update URL params and reset page as currently implemented.
- Do not render an availability/status chip.

### 8. Card container styling and grid density

- Restyle search result cards closer to `frontend-original/` compact card proportions.
- Reduce each card’s visual footprint.
- The search grid should feel like a fuller e-commerce deal grid with more choices visible.
- Cards must have:
  - capped height,
  - uniform card size,
  - consistent cover area,
  - consistent content rhythm.
- Long text must clamp with ellipsis where appropriate:
  - title,
  - author,
  - category if needed.
- The card must not stretch excessively because of long disclaimer/status/content.
- Preserve semantic link behavior to `/book/{id}`.
- Preserve wishlist button behavior.
- Preserve `NƠI BÁN` offer-count label.
- Do not add direct external seller CTAs on search cards.

### 9. Price-drop badge

- Restyle price-drop badge to follow `frontend-original/`:
  - red block,
  - hard black border,
  - hard shadow,
  - slight rotation,
  - placed over the cover,
  - stamped/deal-cue feel.
- Price-drop badge must appear only when the book also has a crossed-out reference/original price.
- That means badge display requires:
  - `book.price_drop` exists,
  - `lowest_eligible_price` is not null,
  - `highest_eligible_price` is not null,
  - `highest_eligible_price > lowest_eligible_price`.
- A book without crossed-out reference price must not show a price-drop badge.
- Badge amount must correspond to current price/reference price relationship, using existing API/schema data.
- Do not invent MSRP.
- Do not add discount percentage.
- Do not add backend fields.

### 10. Card disclaimer

- Remove compact per-card price disclaimer from SearchResultPage cards.
- Do not remove shared `PriceDisclaimer` globally if other pages still need it.
- Prefer a SearchPage-specific card prop or call-site prop so Homepage, WishlistPage, and ProductDetail-related cards are not unintentionally changed.
- Rely on the SearchResultPage bottom disclaimer for this page.

### 11. Bottom disclaimer

- Restyle bottom disclaimer to follow `frontend-original/`.
- Use a more intentional formal-disclosure treatment:
  - top divider,
  - info icon,
  - uppercase microcopy,
  - reduced opacity,
  - compact line height.
- Disclaimer copy must remain Vietnamese.
- Copy must explain:
  - prices are reference prices,
  - prices are recorded from external sellers,
  - users should check again at seller site before buying,
  - DealSach does not sell books directly,
  - DealSach does not handle delivery or returns.
- Do not add affiliate disclosure beside every card/CTA.

### 12. Preserve shared behavior

Because `frontend/src/app/shared.tsx` is shared across Homepage, SearchPage, WishlistPage, ProductDetailPage, and related shelves:

- Do not regress T0022 Homepage card behavior.
- Do not regress T0023 ProductDetail related-card behavior.
- Do not regress WishlistPage card behavior.
- If search-card layout needs adjustment, prefer SearchPage-specific props or a scoped search-card variant.
- Do not globally alter `ApiBookCard` defaults unless all call sites remain visually correct.
- Do not change public API types unless TypeScript build requires a type-only compatibility update.
